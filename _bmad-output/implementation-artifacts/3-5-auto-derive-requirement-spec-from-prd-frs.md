# Story 3.5: Auto-derive RequirementSpec from PRD FRs

Status: ready

## Story

As a platform operator,
I want requirement specs auto-derived from PRD functional requirements and workflow type,
so that every workflow has a quality gate without manual spec authoring.

## Acceptance Criteria

1. Given a new workflow request, when the harness initialises a candidate, then a `RequirementSpec` is generated by mapping PRD FRs to the workflow family type.
2. Given any workflow type, the generated spec always includes the global hard rule: `error_handler_wired: "RumKLiLA2onXkppj"`.
3. Given a workflow family with no existing base template, when spec generation runs, then an LLM call infers intent from node names/prompts and produces a spec with at least structural and error-handler assertions.
4. Given spec generation completes, the spec is written to `test/fixtures/contracts/<slug>.json` and committed to the feature branch.
5. Given a brownfield workflow with no existing contract, when the harness first encounters it, then spec generation runs from the live workflow definition and a new contract is created and registered.

## Implementation Summary

- `RequirementSpec` domain type: `{ version, workflowSlug, features[], assertions[], createdAt, source: "auto-derived"|"authored" }`
- `SpecDeriver` service: maps FR list + workflow family → base assertion set
- Global rule injector: always appends `{ type: "error_handler_check", workflowId: "RumKLiLA2onXkppj" }` regardless of family
- LLM-fallback deriver: when no family template exists, calls cheap LLM with node names/prompt text to infer assertion candidates
- Workflow registry: SQLite table `workflow_registry (workflowId, slug, contractPath, contractVersion, createdAt, updatedAt)`
- Contract writer: writes JSON to `test/fixtures/contracts/<slug>.json`, stages for commit on feature branch
